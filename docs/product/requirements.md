# 旅行规划网页需求文档

## 文档说明

本文档分为两部分：

- 第一部分（1-13）：需求V1，表示项目在正式进入持续迭代前的基线规划需求
- 第二部分（14）：持续迭代需求，表示在基线需求之后，每一轮新增或调整的产品需求版本

维护规则：

- 基线需求主要用于保留项目最初的完整规划，不作为日常频繁更新区域
- 后续每次与你沟通形成的新方向、近期待办、结构调整或技术方案变化，都统一追加到第 14 章
- 第 14 章按需求版本持续递增，例如：`需求V2`、`需求V3`、`需求V4`
- 新增需求版本时，只记录“新增、调整、重排”的内容，不重复抄写整份基线需求

## 第一部分：基线需求（需求V1）

## 1. 项目目标

做一个供个人旅行规划使用的网页工具，用于在出发前整理多日行程、地点顺序、停留时间、交通方式，并在地图上可视化展示每日路线，最后导出 PDF 方便分享。

参考产品：圆周轨迹。

## 2. 目标用户

- 当前主要用户：你自己
- 典型使用场景：
  - 已经确定了出游人数
  - 已经确定了出游日期
  - 已经初步收集到本次旅行可能会去的地点
  - 需要按天规划顺序、时间和交通方式
  - 需要地图可视化结果与可分享的 PDF

## 3. 核心流程

1. 输入本次旅行基础信息
2. 搜索并建立本次旅行的候选地点库
3. 系统根据日期自动生成每日行程容器
4. 将候选地点拖入某一天，并按顺序排列
5. 通过拖动调整同一天内地点顺序
6. 为每个地点设置类型和停留时间
7. 为相邻两个地点之间设置出行方式
8. 系统计算每日总停留时间、路程和预估出行时间
9. 地图展示该日全部地点与路线
10. 导出整趟旅行的 PDF

## 4. 基础信息

当前版本仅保留以下字段：

- 旅行名称
- 出行人数
- 开始日期
- 结束日期

说明：

- 日期范围用于自动生成 Day 1、Day 2、Day 3 等每日区域
- 地图 Key 不在表单中输入，改为单独配置文件管理

## 5. 地点库

每个地点保留以下信息：

- 地点名称
- 完整地址
- 所属城市
- 经度
- 纬度
- POI ID（如可获取）
- 备注

录入方式：

- 用户输入关键词或地址
- 直接在搜索框下拉结果中查看最多 5 条地点结果
- 每条结果右侧提供“添加”按钮，可直接加入地点库
- 城市、地址、经纬度等信息由搜索结果自动填充

地点库特点：

- 一次旅行可录入多个候选地点
- 候选地点可重复加入不同行程
- 同一个地点拖入行程后，生成行程项实例，不影响原始地点库

## 6. 每日行程项设置

每个被拖入某一天的地点块，至少需要配置以下信息：

- 关联地点
- 类型
- 停留时长
- 开始时间（可选，后续扩展）
- 备注

### 6.1 地点类型建议分类

首版采用以下分类：

- 起点
- 终点
- 景点
- 途径点
- 饭店
- 酒店
- 交通枢纽
- 购物
- 休闲
- 备用

### 6.2 行程项排序

- 同一天内，地点块支持拖拽排序
- 支持从候选地点库拖入每日行程
- 后续可扩展为跨天拖拽移动
- 顺序决定地图编号和路线连接顺序

## 7. 出行方式设计

两个相邻地点之间可设置一个出行方式。

首版建议支持：

- 步行
- 打车
- 自驾
- 公交 / 地铁
- 骑行
- 火车
- 飞机
- 轮船
- 其他

## 8. 自动计算逻辑

系统需要根据地点顺序和交通方式，计算：

- 每段路程距离
- 每段预计出行时间
- 每个地点停留时长
- 每日总行程时间
- 每日总路程

首版规则：

- 如果高德 API 可用，优先使用高德路线服务获取步行、驾车、公交、骑行等时间和路程
- 对于火车、飞机、轮船等方式，首版采用估算逻辑
- 如果 API 不可用，则所有方式先按估算规则计算，保证网页仍可使用

## 9. 地图展示需求

地图区域需要支持：

- 展示当日全部地点标记
- 按顺序显示编号
- 区分起点、终点和中间点
- 根据顺序连接路线
- 不同出行方式使用不同颜色
- 估算路径可用虚线显示
- 提供路线图例
- 自动缩放到当日路线范围

## 10. PDF 导出需求

导出内容包括：

- 旅行基础信息
- 地点库
- 每日详细行程
- 每日统计
- 地图视图
- 当前查看日的路线分段摘要

实现策略：

- 首版采用浏览器打印能力导出 PDF
- 提供针对打印场景的专门样式
- 打印时改为纵向分块排版，避免屏幕三栏布局影响阅读

## 11. 数据保存

首版支持：

- 浏览器本地自动保存
- 手动清空重置

## 12. 当前版本功能范围

当前版本实现以下功能：

- 新建旅行基础信息
- 根据日期自动生成多日区域
- 搜索框下拉展示地点结果并直接加入地点库
- 拖动地点到每日行程
- 拖动调整行程顺序
- 设置地点类型
- 设置停留时间
- 设置相邻段出行方式
- 自动估算距离和时间
- 地图展示当日点位和路线
- 本地保存
- PDF 导出

## 13. 高德地图接入方案

### 13.1 配置方式

高德 Key 不再在页面表单中输入，改为通过本地配置文件管理。

建议使用：

- `config.js`：本地实际配置，不提交公开环境时可自行保留
- `config.example.js`：示例配置

配置内容建议包括：

- Web 端 JS API Key
- 安全密钥 `securityJsCode`
- 默认搜索城市

### 13.2 申请方式

需要去高德开放平台申请。

基本流程：

1. 注册并登录高德开放平台开发者账号
2. 在控制台创建应用
3. 为应用添加 Web 端 JS API 的 Key
4. 获取 Key 和安全密钥
5. 将 Key 和安全密钥填入本地 `config.js`

### 13.3 使用建议

- 开发阶段可使用明文方式配置 `securityJsCode`
- 正式生产环境更建议走代理服务器转发方式
- 本地静态页面首版先采用配置文件方式，方便你自己使用

## 第二部分：持续迭代需求

## 14. 持续迭代需求记录

### 14.1 需求V2（2026-03-24）

#### 14.1.1 本轮背景

本轮需求来自项目从“单页静态旅行规划工具”向“支持账号、云端保存和多人协作的旅行规划平台”升级的阶段。重点不是重写基线需求，而是在现有产品基础上补充新的结构、能力和实施顺序。

#### 14.1.2 已确认的新产品方向

- 产品命名方向更新为：`行旅派 | GoPace`
- 产品定位从“单页静态旅行规划工具”逐步升级为“支持账号、云端保存与多人协作的旅行规划平台”
- 前端信息架构调整为双页面结构，并增加顶部菜单栏
- 首页作为账号页使用
- 功能页作为旅行规划页使用

首页设计目标：

- 首次进入时，首页优先展示注册入口
- 用户完成注册后，首页切换为个人信息页
- 个人信息页展示基础账号信息、已保存计划数量、最近编辑计划等概要信息

功能页设计目标：

- 保留当前已有的旅行规划主界面能力
- 用户完成一个规划后，可保存为个人旅行计划
- 已保存的计划支持归档、查看、继续编辑与分享
- 后续为多人共同规划同一行程预留扩展空间

#### 14.1.3 分阶段待办计划

##### P0：后端与账号体系优先落地

- 设计并接入后端服务，替代当前仅依赖浏览器本地存储的模式
- 支持用户注册、登录、退出登录
- 建立用户与旅行计划之间的一对多关系
- 支持旅行计划的创建、保存、更新、删除、归档
- 为后续多人协作预留数据结构，如计划成员、角色、权限字段
- 设计基础鉴权机制，包括登录态保存、接口鉴权、基础安全校验
- 规划从当前 `localStorage` 数据到后端数据模型的兼容方案

##### P1：前端页面结构升级

- 增加顶部菜单栏，首版先保留两个菜单：`首页`、`功能页`
- 引入页面路由或等效的页面切换结构
- 首页未登录状态展示注册表单，并预留登录入口
- 首页登录后展示个人信息页，包括昵称、账号信息、旅行计划列表摘要
- 功能页承接当前规划能力，并与账号体系打通
- 增加“保存计划”“归档计划”“查看历史计划”“继续编辑”相关入口

##### P2：旅行计划数据管理能力

- 一个用户可以拥有多个旅行计划
- 每个旅行计划支持草稿、已完成、已归档等状态
- 提供旅行计划列表页或列表区域，支持查看、筛选和进入详情
- 支持保存后再次打开并恢复完整编辑状态
- 支持基础分享能力设计，如分享链接、只读访问、后续协作入口

##### P3：规划体验增强

- 支持旅行计划的 `JSON` 导入与导出，便于备份、迁移和恢复
- 增加每日统计面板，汇总总路程、总时长、停留时长、预计结束时间
- 增强地点信息，支持预算、人均、营业时间、预约链接、标签等扩展字段
- 支持地点跨天拖拽移动，而不只是在同一天内排序
- 增加行程模板能力，如城市暴走版、亲子轻松版、周末短途版
- 持续优化 PDF 导出效果，使其更接近正式旅行路书

##### P4：多人协作能力

- 支持一个旅行计划邀请多位成员共同参与
- 区分创建者、编辑者、查看者等角色
- 支持多人查看同一计划，并逐步扩展到共同编辑
- 支持计划分享后的权限控制与失效机制
- 视后端实现情况决定是否引入实时同步能力

#### 14.1.4 推荐实施顺序

1. 先完成后端接入与账号体系
2. 再完成菜单栏、首页账号页、功能页的双页面结构
3. 接着打通旅行计划的云端保存、列表、归档与回显
4. 然后补强导入导出、统计面板、模板等体验型功能
5. 最后推进分享与多人共同规划

#### 14.1.5 P0 技术方案建议：Supabase

当前项目是部署在 GitHub Pages 的静态网页，因此第一阶段最合适的后端方案不是先自建一套传统服务端，而是采用：

- 前端静态站点：GitHub Pages
- 认证服务：Supabase Auth
- 数据库：Supabase Postgres
- 文件存储：Supabase Storage（后续可用于头像、行程封面、附件）
- 实时协作预留：Supabase Realtime
- 服务端扩展：Supabase Edge Functions（第二阶段再引入）

首版架构建议：

- 浏览器前端直接使用 Supabase JS 客户端访问 Auth 和数据库
- 浏览器端只使用 `anon key`
- 数据访问安全依赖 Supabase 的 Row Level Security（RLS）策略控制
- 禁止在前端暴露 `service_role key`
- 复杂逻辑如“邀请成员协作”“生成分享链接”“批量归档”“管理员操作”放到后续 Edge Functions 中实现

首版不建议一开始就自建 Node.js 后端，原因包括：

- 当前项目本身是纯静态页面，直接接入 Supabase 成本最低
- 用户注册、登录、会话管理、数据库访问都可以先由 Supabase 官方能力承担
- 你可以更快把“账号 + 云端保存旅行计划”跑通
- 后续如果协作逻辑变复杂，再增加中间层服务也不迟

#### 14.1.6 P0 数据模型建议

为兼容当前前端已经存在的本地 `state` 结构，第一阶段建议采用“元数据字段 + 整体快照 JSON”混合模型，而不是一开始就完全拆成十几张强关系表。

建议首版至少包含以下表：

- `profiles`
  - 作用：保存用户公开资料和扩展信息
  - 关键字段：`id`、`email`、`display_name`、`avatar_url`、`created_at`、`updated_at`
- `trip_plans`
  - 作用：保存一份旅行计划的主记录
  - 关键字段：`id`、`owner_id`、`title`、`status`、`start_date`、`end_date`、`travelers`、`snapshot`、`created_at`、`updated_at`、`archived_at`
- `plan_members`
  - 作用：为多人协作预留成员关系
  - 关键字段：`id`、`plan_id`、`user_id`、`role`、`created_at`

字段说明建议：

- `trip_plans.snapshot` 使用 `jsonb` 保存当前整份前端规划数据
- `status` 首版可用：`draft`、`active`、`archived`
- `owner_id` 关联 `auth.users.id`
- `plan_members` 首版可以先建表但前端暂时不启用，用来给后续多人共同规划留扩展位

这种方案的好处：

- 最贴合你当前前端的本地数据结构，接入快
- 先把“注册账号 + 保存云端计划 + 查看历史计划”打通
- 后续如果要做多人协作、局部编辑、操作记录，再逐步把 `snapshot` 拆分为更细的业务表

#### 14.1.7 P0 实施步骤建议

1. 创建 Supabase 项目
2. 开启 Email + Password 登录
3. 配置 `Site URL` 与 `Redirect URLs`
4. 建立 `profiles`、`trip_plans`、`plan_members` 三张核心表
5. 为三张表开启 RLS 并编写“仅自己可读写”的策略
6. 前端接入 Supabase JS 客户端
7. 完成注册、登录、退出登录
8. 完成“将当前本地规划保存为云端旅行计划”
9. 完成“读取我的旅行计划列表并恢复编辑”
10. 再进入菜单栏和双页面结构改造

#### 14.1.8 安全约束

- 前端公开配置中只能放 `Supabase URL` 和 `anon key`
- `service_role key` 只能用于服务端或 Edge Functions
- 所有业务表必须开启 RLS
- 与用户私有数据相关的增删改查都必须基于 `auth.uid()` 做权限限制
- 邮箱确认、密码重置、分享链接跳转都需要提前配置正式域名与开发域名白名单
### 14.2 需求V3（2026-03-25）

#### 14.2.1 本轮目标

本轮需求重点不再是单点补功能，而是把 GoPace 从“能用的旅行规划器”继续推进成“更像长期使用产品”的状态。核心方向包括：

- 提升首次使用门槛控制，让游客模式和注册登录之间的过渡更顺滑
- 提升规划过程中的信息密度和决策效率，减少来回切页和重复录入
- 提升计划完成后的沉淀价值，让行程库、归档、足迹和分享形成闭环
- 提前补齐适合后续增长的产品骨架，例如模板、协作、预算、提醒和推荐

#### 14.2.2 已确认的新增方向

##### A. 游客模式深化

- 游客模式不只是“能进入功能页”，而是形成完整的轻体验链路
- 游客状态下支持显式保存本地草稿，并提示“登录后可同步到云端”
- 登录成功后支持把游客阶段的本地草稿无感迁移为当前账号下的第一份计划
- 游客模式下首页和功能页增加轻提示，明确哪些能力无需登录，哪些能力需要账号

##### B. 规划效率增强

- 增加“智能起步”入口，例如：
  - 从模板开始
  - 从空白计划开始
  - 从最近一次计划复制开始
- 增加计划内的“按城市 / 按天 / 按主题”三种组织视角，降低长行程编辑压力
- 增加地点快速操作能力，例如：
  - 批量移动到某一天
  - 批量删除
  - 一键复制到其他天
  - 一键设为起点 / 终点
- 增加地点卡片的“待决策”状态，用于标记还没确定是否保留的候选点

##### C. 行程质量辅助

- 增加每日节奏诊断：
  - 当天地点过多提醒
  - 路程过长提醒
  - 连续跨城提醒
  - 早出晚归强度提醒
- 增加时间冲突与逻辑校验：
  - 出发时间晚于到达时间
  - 同一天时间总量明显超载
  - 酒店未设置但跨天连续规划
- 增加“建议优化”面板，用更产品化的方式提示用户调整顺序、删减地点或拆分天数

##### D. 预算与决策信息

- 增加计划级预算模块，支持记录：
  - 总预算
  - 人均预算
  - 交通预算
  - 住宿预算
  - 餐饮预算
  - 门票预算
- 增加地点级费用字段，用于估算单点成本
- 增加每日花费汇总和整趟旅行预算偏差提示
- 后续可扩展为“预算版 PDF / 分享页”

##### E. 行程库升级

- 行程库增加更清晰的卡片分层：
  - 进行中
  - 待完善
  - 已归档
  - 最近浏览
- 增加计划封面、标签和目的地摘要，避免行程库列表过于同质化
- 增加“收藏计划 / 置顶计划 / 最近打开”能力，适合长期积累后快速回到常用计划
- 增加归档后的“旅行总结字段”，例如一句话总结、最佳地点、遗憾清单

##### F. 足迹系统升级

- 足迹地图从“展示去过哪些地方”升级为“旅行记录系统”
- 除省市点亮外，增加：
  - 首次到访时间
  - 到访次数
  - 代表计划
  - 城市照片预留位
- 增加“旅行年度回顾”视角，例如：
  - 今年去了多少城市
  - 最常去的省份
  - 最远的一次旅行
- 增加“未点亮愿望清单”概念，把想去和去过连接起来

##### G. 分享能力升级

- 除 PDF 外，增加可公开访问的分享页
- 分享页至少支持两种模式：
  - 只读完整版
  - 精简展示版
- 分享页可选择是否显示预算、联系方式、备注等敏感字段
- 后续支持生成更适合社交传播的卡片封面和摘要

##### H. 协作能力预留

- 在现有 `plan_members` 预留结构上，进入“轻协作”阶段
- 第一阶段不追求复杂实时协同，而是先支持：
  - 邀请他人查看
  - 邀请他人编辑
  - 标记是谁最后修改了计划
- 后续再扩展评论、建议、版本比较和实时协作

#### 14.2.3 推荐优先级

##### P0：尽快落地

- 游客草稿迁移到登录账号
- 账号中心与游客模式提示完善
- 行程节奏诊断与冲突提醒
- 行程库增加“待完善 / 最近打开 / 置顶”基本能力

##### P1：提升产品完成度

- 预算模块
- 计划模板
- 分享页首版
- 足迹系统的到访次数与代表计划展示

##### P2：形成差异化体验

- 旅行总结卡
- 年度旅行回顾
- 愿望清单与足迹联动
- 建议优化面板

##### P3：面向增长预留

- 协作邀请
- 编辑痕迹
- 评论与建议
- 更细的权限和版本比较

#### 14.2.4 设计与体验要求

- 新增能力不能破坏当前“首页 / 行程库 / 功能页”的信息结构
- 游客模式相关入口必须明确，但不能让首页重新变回信息拥挤状态
- 预算、总结、愿望清单等新增信息需要有独立的信息层级，不与基础规划字段混写
- 如果后续加入更多面板，优先考虑抽屉、分段区域或二级页，而不是无限堆叠在主工作区

#### 14.2.5 建议实施顺序

1. 先做游客草稿迁移、右上角状态统一、账号提示完善，打磨首次使用链路
2. 再做行程节奏诊断、时间冲突提醒和待完善状态，提升规划过程质量
3. 接着补预算模块、模板能力和分享页首版，形成更完整的产品闭环
4. 最后推进旅行总结、年度回顾、愿望清单和轻协作，拉开与普通规划工具的差异

### 14.3 需求V4（2026-03-26）

#### 14.3.1 本轮背景

当前产品已经形成 `首页 / 行程库 / 功能页` 三页结构，但地点管理仍然主要依附在规划页左侧候选区中，导致“日常收藏地点”和“正式排布行程”两类动作耦合过深。

本轮目标是把“先收藏地点，再进入规划”的使用路径独立出来，同时把原有“行程库”进一步转为更接近“我的 / 个人中心”的页面结构。

#### 14.3.2 信息架构调整

- 顶部导航升级为四页结构：
  - 首页
  - 地点库
  - 规划页
  - 个人信息页
- 原“功能页”正式更名为 `规划页`
- 原“行程库”不再作为单独的第二页入口，而是并入 `个人信息页` 作为其中的重要模块
- `个人信息页` 后续用于继续扩展好友、共同收藏、偏好设置等“我的”能力

#### 14.3.3 地点库页需求

地点库页定位为：用户在正式规划之前，先长期沉淀自己想去、想吃、想住的地点收藏库。

核心目标：

- 支持日常看到、听到、临时想到的地点先行收藏
- 支持在真正开始规划某次旅行时，直接从个人地点库调取已有地点
- 让地点管理从规划页左侧候选区中独立出来，形成更强的单独页面

地点库页至少需要支持：

- 添加地点
- 查看已收藏地点列表
- 对地点进行分类管理，首轮分类包含：
  - 玩
  - 吃
  - 住
  - 其他
- 对地点进行省份维度切换与筛选
- 与规划页共享同一批地点数据，保证在地点库中整理后，可以直接在规划页中调用

#### 14.3.4 规划页调整要求

- 规划页继续承担旅行名称、日期、多日行程编排、路线预览和云端保存等核心规划能力
- 原本左侧“候选地点库”改为“来自地点库的候选地点区”
- 规划页不再承担完整地点管理职责，而是承担：
  - 快速查看已收藏地点
  - 从地点库拖入每天行程
  - 在需要时跳转到地点库继续补充和整理

#### 14.3.5 个人信息页定位

- 个人信息页作为新的“我的”页面
- 当前阶段至少承接：
  - 账号信息概览
  - 地点收藏概况
  - 行程计划管理能力
- 原有行程库的完整计划管理能力保留在本页中
- 后续在本页继续扩展：
  - 加好友
  - 社交关系
  - 共同收藏
  - 个人偏好与更多个人化入口

#### 14.3.6 建议实施顺序

1. 先完成导航和页面信息架构重排，明确四页结构
2. 再完成地点库首版页面，支持添加、分类和省份切换
3. 接着调整规划页左侧区域，使其成为地点库的调用入口，而不是完整管理页
4. 最后把原行程库整合进个人信息页，为后续“我的”能力继续留位

### 14.4 需求V5（2026-03-27）

#### 14.4.1 背景

当前地点库中的地点主要保存在浏览器本地，用户在游客状态下先收藏地点后，再登录账号时，地点库无法自然延续到账号体系内；换设备或清理浏览器缓存后，地点库也无法跟随账号回读，导致地点沉淀能力断层。

#### 14.4.2 目标

- 让地点库具备账号级云端持久化能力
- 让用户登录后可以直接看到当前账号已保存的地点
- 让游客状态下先添加的地点，在登录后可以并入当前账号的地点库
- 让地点库与具体某一份行程解耦，不因切换计划而被覆盖

#### 14.4.3 需求拆解

##### A. 账号级地点库存储

- Supabase 需要新增地点库云端快照字段，用于保存当前账号全部已收藏地点
- 字段需要支持空数组默认值，避免老账号因字段缺失导致读取失败
- 地点库云端数据归属账号本人，不与其他账号共享

##### B. 登录回读与自动合并

- 用户登录后，前端需要先读取当前账号的地点库云端快照
- 如果本地地点库与云端地点库同时存在，系统应自动合并，而不是简单覆盖
- 合并时需要具备去重能力，避免同一地点因游客收藏和账号已有数据重复出现

##### C. 地点库操作自动同步

- 在地点库新增地点后，应自动同步到云端
- 在地点库编辑分类、名称或其他基础信息后，应自动同步到云端
- 在地点库删除地点后，应自动同步到云端
- 同步失败时要给出明确提示，避免用户误以为已经成功入云

##### D. 与行程计划解耦

- 打开某份云端计划时，不应把账号级地点库替换成该计划快照内的地点集合
- 新建空白计划、重置当前规划或切换页面时，不应清空已登录账号的地点库
- 行程内引用地点仍可继续使用地点库中的同一份地点数据

#### 14.4.4 验收标准

- 用户在未登录状态下添加地点后登录，同一账号下仍能看到这些地点
- 用户登录后新增、编辑、删除地点，刷新页面后仍可回读
- 用户重新登录同一账号或在另一设备登录后，可以看到该账号已保存的地点库
- 打开已有计划、新建计划、切换计划后，地点库不会异常丢失或回退

### 14.5 需求V6（2026-03-27）
#### 14.5.1 背景

当前个人信息页已经承接账号信息、地点收藏概况和行程管理，但还缺少真正能把“我和谁一起出发”串起来的社交层。用户需要的不是泛社交广场，而是围绕旅行搭子展开的轻关系能力：能加好友、能把一条行程发给对方、也能围绕路线继续聊细节。

#### 14.5.2 本轮目标

- 在个人信息页落地一版好友中心
- 支持搜索推荐对象并发起好友申请
- 支持处理收到的好友申请
- 支持在好友之间打开私信会话
- 支持把当前行程直接分享给好友
- 支持在个人页沉淀最近的分享动态

#### 14.5.3 本轮实现边界

- 首轮以个人页中的社交工作台为主，不新增独立社交页
- 首轮先使用账号维度的本地持久化验证交互链路，不直接承诺跨设备同步
- 分享能力先聚焦“把当前绑定行程或当前规划草稿发给好友”
- 私信能力先支持文本消息和“分享一条行程”两种消息类型
- 不在这一轮加入群聊、评论、朋友圈式时间线、公开主页或复杂权限

#### 14.5.4 页面结构要求

- 个人信息页顶部保留账号概况与好友中心总览
- 中部新增三栏社交工作台：
  - 左侧：好友网络，包括推荐对象、待处理申请、好友列表
  - 中部：私信会话，包括会话切换、消息流、发送消息、分享当前行程
  - 右侧：分享动态，聚合最近发出的和收到的行程分享
- 整体视觉继续沿用当前工作台风格，不做成独立聊天软件界面

#### 14.5.5 交互要求

- 搜索好友时可按昵称、邮箱、城市或标签检索推荐对象
- 发出申请后，该对象应从推荐区转入待处理状态
- 接受申请后，应自动生成与该好友的私信会话
- 点击好友列表中的好友，可直接打开对应私信
- 在私信中点击“分享当前行程”后，应同步写入会话消息和分享动态
- 当前没有云端计划时，仍允许把当前规划草稿作为分享对象

#### 14.5.6 后续云端化预留

- 后续接入 Supabase 时，需要将好友关系、好友申请、私信线程、消息记录和行程分享拆成独立表
- 好友与私信默认只对关系双方可见
- 行程分享应与 `trip_plans` 建立引用，但需要保留分享时的标题快照，避免原计划改名后历史消息失真

#### 14.5.7 验收标准

- 用户可以在个人页看到推荐好友，并成功发出申请
- 用户可以接受或忽略好友申请，好友列表会同步变化
- 用户可以打开私信会话并发送消息
- 用户可以把当前行程分享给某个好友，并在会话与分享动态中同时看到记录
- 刷新页面后，当前账号在本机上的好友、私信和分享状态仍能保留

### 14.6 需求V7（2026-03-27）
#### 14.6.1 背景

当前好友、私信和行程分享模块已经落在个人信息页中，导致“我的资产管理”和“社区互动”两类任务混在同一页里，页面焦点不够清晰，也不利于后续继续扩展社区能力。

#### 14.6.2 调整目标

- 新增独立社区页，专门承载好友、私信、分享动态等社区能力
- 原个人信息页回归账号概况、地点收藏、足迹沉淀和旅行计划管理
- 顶部导航升级为 `首页 / 地点库 / 规划页 / 社区页 / 个人信息` 五页结构
- 社区页在导航顺序上位于个人信息页之前

#### 14.6.3 页面边界

- 好友中心总览、好友网络、私信对话、分享动态从个人信息页迁移到社区页
- 个人信息页不再继续承载上述社区模块，避免“我的”与“社交”信息混杂
- 社区页继续保持登录后可访问的边界，与当前账号体系一致
- 当前阶段继续复用本地持久化的社区数据模型，不额外引入新的后端结构

#### 14.6.4 验收标准

- 登录用户可以从顶部导航进入独立社区页
- 社区页能正常查看好友、申请、私信和分享动态
- 个人信息页仍可正常查看账号概况并管理旅行计划
- 社区页在顶部导航中的顺序位于个人信息页前

### 14.7 需求V8（2026-03-27）
#### 14.7.1 背景

当前地点库虽然已经接入 Supabase 同步，但前端仍以本地状态为主，导致用户很难判断“这些地点是否已经真正绑定到当前账号”。当切换设备、重新登录或回到同一账号时，地点库的主存储边界需要更清晰。

#### 14.7.2 目标

- 让登录状态下的地点库以 Supabase 云端为主数据源
- 让游客模式下的地点库与登录后的账号地点库边界更清晰
- 让用户可以直接看到地点库当前是“本地缓存”还是“账号云端”，以及最近一次同步时间

#### 14.7.3 需求拆解
##### A. 存储模式

- 游客模式下，地点库继续使用当前浏览器本地存储
- 登录后，地点库应以 `profiles.place_library_snapshot` 作为当前账号的权威数据源
- 登录状态下本地存储只用于缓存或游客迁移，不再作为登录后地点库的主数据源

##### B. 游客地点迁移

- 当前浏览器中的游客地点库可以在用户第一次登录同一账号时并入到云端
- 合并后需要记录该浏览器已对该账号执行过这次游客地点迁移，避免后续每次登录都重复合并
- 合并过程需要具备去重能力，避免重复地点再次入云

##### C. 同步可视化

- 地点库页需要明确标识当前是“本地版”还是“云端版”
- 登录状态下，页面需要可以显示云端同步中、同步失败或最近一次同步时间
- 同步失败时需继续给出明确错误提示，避免用户误以为地点已经完全入云

#### 14.7.4 验收标准

- 用户登录后，地点库会从当前账号的云端快照回读，不再以本地缓存作为主来源
- 用户在同一账号下新增、删除和调整分类后，可以在其他设备或刷新后进行回读
- 地点库页可以直接看到当前存储模式与最近一次云端同步时间
