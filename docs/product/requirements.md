## 文档说明

本文档分为两部分：

- 第一部分（1-13）：需求V1，表示项目在正式进入持续迭代前的基线规划需求
- 第二部分（14）：持续迭代需求，表示在基线需求之后，每一轮新增或调整的产品需求版本

维护规则：

- 基线需求主要用于保留项目最初的完整规划，不作为日常频繁更新区域
- 后续每次与你沟通形成的新方向、近期待办、结构调整或技术方案变化，都统一追加到第 14 章
- 第 14 章按需求版本持续递增，例如：`需求V2`、`需求V3`、`需求V4`
- 新增需求版本时，只记录“新增、调整、重排”的内容，不重复抄写整份基线需求

## 第一部分：基线需求（需求V1）

## 1. 项目目标

做一个供个人旅行规划使用的网页工具，用于在出发前整理多日行程、地点顺序、停留时间、交通方式，并在地图上可视化展示每日路线，最后导出 PDF 方便分享。

参考产品：圆周轨迹。

## 2. 目标用户

- 当前主要用户：你自己
- 典型使用场景：
  - 已经确定了出游人数
  - 已经确定了出游日期
  - 已经初步收集到本次旅行可能会去的地点
  - 需要按天规划顺序、时间和交通方式
  - 需要地图可视化结果与可分享的 PDF

## 3. 核心流程

1. 输入本次旅行基础信息
2. 搜索并建立本次旅行的候选地点库
3. 系统根据日期自动生成每日行程容器
4. 将候选地点拖入某一天，并按顺序排列
5. 通过拖动调整同一天内地点顺序
6. 为每个地点设置类型和停留时间
7. 为相邻两个地点之间设置出行方式
8. 系统计算每日总停留时间、路程和预估出行时间
9. 地图展示该日全部地点与路线
10. 导出整趟旅行的 PDF

## 4. 基础信息

当前版本仅保留以下字段：

- 旅行名称
- 出行人数
- 开始日期
- 结束日期

说明：

- 日期范围用于自动生成 Day 1、Day 2、Day 3 等每日区域
- 地图 Key 不在表单中输入，改为单独配置文件管理

## 5. 地点库

每个地点保留以下信息：

- 地点名称
- 完整地址
- 所属城市
- 经度
- 纬度
- POI ID（如可获取）
- 备注

录入方式：

- 用户输入关键词或地址
- 直接在搜索框下拉结果中查看最多 5 条地点结果
- 每条结果右侧提供“添加”按钮，可直接加入地点库
- 城市、地址、经纬度等信息由搜索结果自动填充

地点库特点：

- 一次旅行可录入多个候选地点
- 候选地点可重复加入不同行程
- 同一个地点拖入行程后，生成行程项实例，不影响原始地点库

## 6. 每日行程项设置

每个被拖入某一天的地点块，至少需要配置以下信息：

- 关联地点
- 类型
- 停留时长
- 开始时间（可选，后续扩展）
- 备注

### 6.1 地点类型建议分类

首版采用以下分类：

- 起点
- 终点
- 景点
- 途径点
- 饭店
- 酒店
- 交通枢纽
- 购物
- 休闲
- 备用

### 6.2 行程项排序

- 同一天内，地点块支持拖拽排序
- 支持从候选地点库拖入每日行程
- 后续可扩展为跨天拖拽移动
- 顺序决定地图编号和路线连接顺序

## 7. 出行方式设计

两个相邻地点之间可设置一个出行方式。

首版建议支持：

- 步行
- 打车
- 自驾
- 公交 / 地铁
- 骑行
- 火车
- 飞机
- 轮船
- 其他

## 8. 自动计算逻辑

系统需要根据地点顺序和交通方式，计算：

- 每段路程距离
- 每段预计出行时间
- 每个地点停留时长
- 每日总行程时间
- 每日总路程

首版规则：

- 如果高德 API 可用，优先使用高德路线服务获取步行、驾车、公交、骑行等时间和路程
- 对于火车、飞机、轮船等方式，首版采用估算逻辑
- 如果 API 不可用，则所有方式先按估算规则计算，保证网页仍可使用

## 9. 地图展示需求

地图区域需要支持：

- 展示当日全部地点标记
- 按顺序显示编号
- 区分起点、终点和中间点
- 根据顺序连接路线
- 不同出行方式使用不同颜色
- 估算路径可用虚线显示
- 提供路线图例
- 自动缩放到当日路线范围

## 10. PDF 导出需求

导出内容包括：

- 旅行基础信息
- 地点库
- 每日详细行程
- 每日统计
- 地图视图
- 当前查看日的路线分段摘要

实现策略：

- 首版采用浏览器打印能力导出 PDF
- 提供针对打印场景的专门样式
- 打印时改为纵向分块排版，避免屏幕三栏布局影响阅读

## 11. 数据保存

首版支持：

- 浏览器本地自动保存
- 手动清空重置

## 12. 当前版本功能范围

当前版本实现以下功能：

- 新建旅行基础信息
- 根据日期自动生成多日区域
- 搜索框下拉展示地点结果并直接加入地点库
- 拖动地点到每日行程
- 拖动调整行程顺序
- 设置地点类型
- 设置停留时间
- 设置相邻段出行方式
- 自动估算距离和时间
- 地图展示当日点位和路线
- 本地保存
- PDF 导出

## 13. 高德地图接入方案

### 13.1 配置方式

高德 Key 不再在页面表单中输入，改为通过本地配置文件管理。

建议使用：

- `config.js`：本地实际配置，不提交公开环境时可自行保留
- `config.example.js`：示例配置

配置内容建议包括：

- Web 端 JS API Key
- 安全密钥 `securityJsCode`
- 默认搜索城市

### 13.2 申请方式

需要去高德开放平台申请。

基本流程：

1. 注册并登录高德开放平台开发者账号
2. 在控制台创建应用
3. 为应用添加 Web 端 JS API 的 Key
4. 获取 Key 和安全密钥
5. 将 Key 和安全密钥填入本地 `config.js`

### 13.3 使用建议

- 开发阶段可使用明文方式配置 `securityJsCode`
- 正式生产环境更建议走代理服务器转发方式
- 本地静态页面首版先采用配置文件方式，方便你自己使用

## 第二部分：持续迭代需求

## 14. 持续迭代需求记录

本章节用于记录基线需求（需求V1）之后的持续迭代需求。

追加规则：

- `14.1` 为 `需求V2`
- 后续继续使用 `14.2`、`14.3`、`14.4` 的方式向后追加
- 每个需求版本保留当轮沟通后确认的新增方向与实施重点
- 即使后续已经实现，也保留在这里，作为需求演进轨迹

### 14.1 需求V2（2026-03-24）

#### 14.1.1 本轮背景

本轮需求来自项目从“单页静态旅行规划工具”向“支持账号、云端保存和多人协作的旅行规划平台”升级的阶段。重点不是重写基线需求，而是在现有产品基础上补充新的结构、能力和实施顺序。

#### 14.1.2 已确认的新产品方向

- 产品命名方向更新为：`行旅派 | GoPace`
- 产品定位从“单页静态旅行规划工具”逐步升级为“支持账号、云端保存与多人协作的旅行规划平台”
- 前端信息架构调整为双页面结构，并增加顶部菜单栏
- 首页作为账号页使用
- 功能页作为旅行规划页使用

首页设计目标：

- 首次进入时，首页优先展示注册入口
- 用户完成注册后，首页切换为个人信息页
- 个人信息页展示基础账号信息、已保存计划数量、最近编辑计划等概要信息

功能页设计目标：

- 保留当前已有的旅行规划主界面能力
- 用户完成一个规划后，可保存为个人旅行计划
- 已保存的计划支持归档、查看、继续编辑与分享
- 后续为多人共同规划同一行程预留扩展空间

#### 14.1.3 分阶段待办计划

##### P0：后端与账号体系优先落地

- 设计并接入后端服务，替代当前仅依赖浏览器本地存储的模式
- 支持用户注册、登录、退出登录
- 建立用户与旅行计划之间的一对多关系
- 支持旅行计划的创建、保存、更新、删除、归档
- 为后续多人协作预留数据结构，如计划成员、角色、权限字段
- 设计基础鉴权机制，包括登录态保存、接口鉴权、基础安全校验
- 规划从当前 `localStorage` 数据到后端数据模型的兼容方案

##### P1：前端页面结构升级

- 增加顶部菜单栏，首版先保留两个菜单：`首页`、`功能页`
- 引入页面路由或等效的页面切换结构
- 首页未登录状态展示注册表单，并预留登录入口
- 首页登录后展示个人信息页，包括昵称、账号信息、旅行计划列表摘要
- 功能页承接当前规划能力，并与账号体系打通
- 增加“保存计划”“归档计划”“查看历史计划”“继续编辑”相关入口

##### P2：旅行计划数据管理能力

- 一个用户可以拥有多个旅行计划
- 每个旅行计划支持草稿、已完成、已归档等状态
- 提供旅行计划列表页或列表区域，支持查看、筛选和进入详情
- 支持保存后再次打开并恢复完整编辑状态
- 支持基础分享能力设计，如分享链接、只读访问、后续协作入口

##### P3：规划体验增强

- 支持旅行计划的 `JSON` 导入与导出，便于备份、迁移和恢复
- 增加每日统计面板，汇总总路程、总时长、停留时长、预计结束时间
- 增强地点信息，支持预算、人均、营业时间、预约链接、标签等扩展字段
- 支持地点跨天拖拽移动，而不只是在同一天内排序
- 增加行程模板能力，如城市暴走版、亲子轻松版、周末短途版
- 持续优化 PDF 导出效果，使其更接近正式旅行路书

##### P4：多人协作能力

- 支持一个旅行计划邀请多位成员共同参与
- 区分创建者、编辑者、查看者等角色
- 支持多人查看同一计划，并逐步扩展到共同编辑
- 支持计划分享后的权限控制与失效机制
- 视后端实现情况决定是否引入实时同步能力

#### 14.1.4 推荐实施顺序

1. 先完成后端接入与账号体系
2. 再完成菜单栏、首页账号页、功能页的双页面结构
3. 接着打通旅行计划的云端保存、列表、归档与回显
4. 然后补强导入导出、统计面板、模板等体验型功能
5. 最后推进分享与多人共同规划

#### 14.1.5 P0 技术方案建议：Supabase

当前项目是部署在 GitHub Pages 的静态网页，因此第一阶段最合适的后端方案不是先自建一套传统服务端，而是采用：

- 前端静态站点：GitHub Pages
- 认证服务：Supabase Auth
- 数据库：Supabase Postgres
- 文件存储：Supabase Storage（后续可用于头像、行程封面、附件）
- 实时协作预留：Supabase Realtime
- 服务端扩展：Supabase Edge Functions（第二阶段再引入）

首版架构建议：

- 浏览器前端直接使用 Supabase JS 客户端访问 Auth 和数据库
- 浏览器端只使用 `anon key`
- 数据访问安全依赖 Supabase 的 Row Level Security（RLS）策略控制
- 禁止在前端暴露 `service_role key`
- 复杂逻辑如“邀请成员协作”“生成分享链接”“批量归档”“管理员操作”放到后续 Edge Functions 中实现

首版不建议一开始就自建 Node.js 后端，原因包括：

- 当前项目本身是纯静态页面，直接接入 Supabase 成本最低
- 用户注册、登录、会话管理、数据库访问都可以先由 Supabase 官方能力承担
- 你可以更快把“账号 + 云端保存旅行计划”跑通
- 后续如果协作逻辑变复杂，再增加中间层服务也不迟

#### 14.1.6 P0 数据模型建议

为兼容当前前端已经存在的本地 `state` 结构，第一阶段建议采用“元数据字段 + 整体快照 JSON”混合模型，而不是一开始就完全拆成十几张强关系表。

建议首版至少包含以下表：

- `profiles`
  - 作用：保存用户公开资料和扩展信息
  - 关键字段：`id`、`email`、`display_name`、`avatar_url`、`created_at`、`updated_at`
- `trip_plans`
  - 作用：保存一份旅行计划的主记录
  - 关键字段：`id`、`owner_id`、`title`、`status`、`start_date`、`end_date`、`travelers`、`snapshot`、`created_at`、`updated_at`、`archived_at`
- `plan_members`
  - 作用：为多人协作预留成员关系
  - 关键字段：`id`、`plan_id`、`user_id`、`role`、`created_at`

字段说明建议：

- `trip_plans.snapshot` 使用 `jsonb` 保存当前整份前端规划数据
- `status` 首版可用：`draft`、`active`、`archived`
- `owner_id` 关联 `auth.users.id`
- `plan_members` 首版可以先建表但前端暂时不启用，用来给后续多人共同规划留扩展位

这种方案的好处：

- 最贴合你当前前端的本地数据结构，接入快
- 先把“注册账号 + 保存云端计划 + 查看历史计划”打通
- 后续如果要做多人协作、局部编辑、操作记录，再逐步把 `snapshot` 拆分为更细的业务表

#### 14.1.7 P0 实施步骤建议

1. 创建 Supabase 项目
2. 开启 Email + Password 登录
3. 配置 `Site URL` 与 `Redirect URLs`
4. 建立 `profiles`、`trip_plans`、`plan_members` 三张核心表
5. 为三张表开启 RLS 并编写“仅自己可读写”的策略
6. 前端接入 Supabase JS 客户端
7. 完成注册、登录、退出登录
8. 完成“将当前本地规划保存为云端旅行计划”
9. 完成“读取我的旅行计划列表并恢复编辑”
10. 再进入菜单栏和双页面结构改造

#### 14.1.8 安全约束

- 前端公开配置中只能放 `Supabase URL` 和 `anon key`
- `service_role key` 只能用于服务端或 Edge Functions
- 所有业务表必须开启 RLS
- 与用户私有数据相关的增删改查都必须基于 `auth.uid()` 做权限限制
- 邮箱确认、密码重置、分享链接跳转都需要提前配置正式域名与开发域名白名单
